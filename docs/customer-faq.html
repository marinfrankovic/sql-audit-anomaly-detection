Architecture

What exactly is deployed?

A small, self-contained environment in your Azure subscription: a central log store (Azure Log Analytics), a demo SQL Server (on an Azure VM for the PoC), auditing configured to stream every data access into the log store, a Key Vault for secrets, an Action Group + alert rules for email notifications, and one Azure Workbook dashboard. Optional add-ons: Microsoft Sentinel/UEBA and an Azure OpenAI “AI Analyst”.

How does data flow through the system?

Database access → SQL Audit writes an audit record → the Azure Monitor Agent forwards it (per a Data Collection Rule) into Log Analytics → the three detection layers analyse it → results appear on the dashboard and are pushed as email alerts. Nothing is installed inside the database engine itself.

Does it touch or slow down our production databases?

No. It reads the audit trail, not the data. SQL Server auditing has minimal overhead, and all analysis happens in Log Analytics — away from the database. In the PoC everything is synthetic; in production it would attach to your existing audit stream.

Is this one-off, or continuous?

Continuous. Audit data streams in real time; the ML baseline is recalculated on a rolling window; alerts evaluate every few minutes. The 90 days of history in the PoC is preloaded so the dashboard is populated on day one.

Detection & the three layers

Why three layers instead of just rules?

Rules only catch what you already thought to write down. Attackers and insiders do new things. So we combine: Layer 1 rules for known-bad patterns, Layer 2 machine learning for never-seen-before behaviour, and Layer 3 AI to explain findings in plain language.

What is Layer 1 (deterministic rules)?

Fixed, explainable KQL rules: break-glass account use, DELETE/DROP/GRANT on financial objects, permission escalation, after-hours sensitive-data access, failed-login bursts, schema tampering. Each fires predictably and is fully auditable.

What is Layer 2 (machine learning)?

KQL machine-learning functions (series_decompose_anomalies) learn each user’s and system’s normal pattern over 90 days and flag statistical outliers — e.g. a sudden spike in query volume — with no fixed threshold. This catches behaviour no rule targets, including from privileged users.

What is Layer 3 (AI)?

An Azure OpenAI model reads the flagged evidence and writes a short, plain-language explanation of why it is suspicious and what to investigate first. It is read-only, grounded strictly in the audit records, and never invents events or executes anything.

Does it work for privileged / admin users?

Yes — this is a core strength. Every user is compared to their own baseline, so a legitimate DBA doing something out-of-character (wrong time, wrong data, unusual volume) still stands out even though their access is technically allowed.

How are false positives handled?

Layer 1 rules are precise by design. Layer 2 uses a tunable sensitivity so you can trade off noise vs. coverage. Every alert carries full evidence and an AI explanation, so triage is fast and analysts can confirm or dismiss quickly. Baselines improve as more history accrues.

Cost

What drives the cost?

Primarily Log Analytics data ingestion & retention (priced per GB). Secondary: the demo VM (not needed in production if you use real servers), optional Azure OpenAI (per token — tiny, since only anomalies are analysed), and optional Microsoft Sentinel (per GB) if you enable it.

Roughly how much for the PoC?

The PoC is intentionally small and cost-conscious — a single low-tier VM, modest log volume, and a small AI model (gpt-4o-mini). It is designed to be torn down with one command (azd down --purge) so it only costs while it runs.

How do we control cost in production?

  • Audit only the objects that matter (sensitive tables), not everything.
  • Use Log Analytics commitment tiers and data retention/archival policies.
  • Send only audit metadata — never the underlying data — keeping volumes low.
  • Run the AI layer only on anomalies, not on every event.

Security & data protection

Where does the data live?

Entirely inside your Azure tenant and chosen region. No data is sent to third parties. The AI layer uses your Azure OpenAI resource (your tenant), not a public service.

Is the AI a data-exfiltration risk?

No. The AI Analyst is read-only and receives only the specific audit evidence for a flagged anomaly. It cannot query databases, change anything, or send data outside your tenant, and it is instructed to never invent events.

How are secrets and identities handled?

Secrets are stored in Azure Key Vault; services authenticate with managed identities and least-privilege RBAC where possible. The audit data itself contains access metadata (who/what/when), not card numbers or balances.

Is any real customer data used?

No. The PoC uses 100% synthetic data. No real personal or financial information is present anywhere.

PoC vs production: the PoC favours simplicity (public network access on the demo VM, secrets in Key Vault but no Private Endpoints). A production rollout would add Private Link, network isolation, and hardened RBAC — documented in the post-deployment guide.

On-premises SQL Server & Azure Arc

Does this work with our on-premises SQL Servers?

Yes. The same three-layer analytics work on any SQL Server whose audit stream reaches Log Analytics. The bridge for on-premises (or other-cloud) servers is Azure Arc.

How does Azure Arc collect the data?

  • Install the Azure Arc agent (azcmagent) on the on-prem server — it now appears in Azure as an Arc-enabled server.
  • Deploy the Azure Monitor Agent extension to it and attach a Data Collection Rule that gathers the SQL Server Audit events (from the Windows event log / audit files).
  • Those audit records flow to the same Log Analytics workspace — and the same rules, ML and AI apply, side-by-side with cloud data.

What actually leaves our premises?

Only the audit log records (who accessed what, when, from where) — not the database contents. All agent traffic is outbound HTTPS (443) to Azure; Azure never connects back into your network. It can be routed over Azure Private Link / ExpressRoute to keep the data off the public internet (a few control-plane endpoints, such as Microsoft Entra ID and Azure Resource Manager, may still use public endpoints).

What is “Arc-enabled SQL Server” and do we need it?

Arc-enabled SQL Server extends Arc to the SQL instance, unlocking inventory, best-practices assessment, and Microsoft Defender for SQL. It is complementary: you can collect audit data with just the Arc agent + Azure Monitor Agent, and optionally add Arc-enabled SQL Server for deeper posture management.

What are the prerequisites and overhead on-prem?

A lightweight agent per server, outbound 443 (or Private Link), and permission to onboard. The agent footprint is small; SQL auditing overhead is minimal. No inbound firewall changes and no data-plane access are required.

Can we mix cloud and on-prem in one view?

Yes. Azure SQL, SQL on Azure VMs, and Arc-connected on-prem servers all feed the same workspace, so the dashboard, alerts, ML baselines and AI explanations span your entire estate in one place.
