Contoso Bank SQL Audit & AI Behavior Analytics

Proof of Concept — Facilitator & Demo Guide
Who accessed which database records, when, from where — and was it normal or abnormal?
Azure-native · Deploys with azd up · 90 days of history preloaded · Demo-ready immediately

1 — What you are proving

The message to the customer: “We can see who accessed what database records, when, and from where — and whether the behaviour looks normal or abnormal, even when the user has legitimate elevated access. And an AI layer explains why something is suspicious.”

Everything runs inside Azure. There is one pretend bank database (fake customers, accounts, money), one central logbook (Azure Log Analytics) that records every database access, and one dashboard (an Azure Workbook) that turns the logbook into a story. No real personal data is used anywhere.

2 — The three detection layers (the core message)

This is the point to land with the customer: detection is not just a fixed set of rules. There are three complementary layers, each stronger than the last.

LAYER 1

Deterministic rules

Fixed, explainable KQL rules. Example: “break-glass account used”, “DELETE on a wire-transfer table”, “sensitive data read after hours”. Each rule either fires or it doesn’t.

Great for known-bad patterns and compliance. Predictable, zero false-negatives on the rule itself.

LAYER 2

Machine learning

KQL ML (series_decompose_anomalies) learns each user’s and system’s normal behaviour over 90 days and flags statistical outliers — e.g. a sudden spike in query volume — with no fixed threshold.

Catches the “never-seen-before” behaviour that no rule was written for.

LAYER 3

AI explanation

Azure OpenAI reads the flagged evidence and explains in plain language why it is suspicious and what to investigate first. It is read-only and grounded strictly in the audit records — it never invents events.

Turns raw signals into an analyst-ready narrative.

How to say it: “Layer 1 catches what we know is bad. Layer 2 uses machine learning to catch what we’ve never seen by learning normal behaviour. Layer 3 uses AI to explain it so an analyst can act in seconds — not hours.”

4 — One-time setup (before the meeting)

From the root of the project folder:

azd auth login
azd up            # builds all Azure resources AND preloads 90 days of history

cd scripts
./run-poc-scenarios.ps1     # setup + mock data + normal activity + WOW detections + AI

run-poc-scenarios.ps1 is the “head waiter” — it runs the other scripts in the right order. You do not need to run the individual scripts yourself.

Tip: After it finishes, open the workbook and set Time range = 30 days. Every panel should already be full. If a panel looks empty, widen the Time range to 30 or 90 days.

5 — Running the demo (about 30 minutes)

0. Open the dashboard. Azure Portal → the resource group rg-sqlaudit-demo → open the workbook “Contoso Bank SQL Audit & AI Behavior Analytics PoC”. Set Time range = 30 days. There is only one workbook — nothing else to pick.
1. Executive Overview. Lead with the tiles: total audit events, total anomalies, high-risk anomalies, break-glass usage. “This is 90 days of who-did-what across the bank’s databases.”
2. Timeline & ML baseline Layer 2. Point at the right-hand chart. “Machine learning has learned the normal daily volume. These highlighted points are statistical anomalies — no one wrote a rule for them.”
3. User Behavior Analytics. Show privileged vs standard users and who touches the most distinct objects. “Even a legitimate DBA is measured against their own normal.”
4. Layer 1 — Deterministic Rule Detections Layer 1. Walk the six tiles: break-glass used, DBA after-hours sensitive access, suspicious DELETE on a financial object, permission escalation, out-of-role access, query-volume anomaly. “These are the known-bad patterns, each explainable.”
5. Investigation View. Sort by AnomalyScore (descending). “One place to triage — the riskiest behaviour floats to the top, with the full evidence: user, time, object, statement, source IP.”
6. Layer 3 — AI-Assisted Findings Layer 3. Read one AI explanation aloud. “The AI reads the same evidence and tells the analyst, in plain language, why it’s suspicious and what to check first — grounded only in the audit log, never invented.”
7. (Optional) Show it live. In a terminal run ./generate-wow-detections.ps1 -RunAi, then refresh the workbook to show new activity flowing in.

6 — Likely questions

QuestionAnswer
“Is this just rules?”No — three layers: deterministic rules (L1), machine-learning behavioural baselines (L2), and AI explanation (L3). Rules alone miss novel behaviour; ML catches it.
“Does it work for privileged users?”Yes. Every user is compared to their own normal, so a legitimate DBA doing something out-of-character still stands out.
“Does the AI make things up?”No. It is read-only and grounded strictly in the audit evidence; it cites user, time, object, statement, and IP.
“Is this real data?”All synthetic. The 90-day history is generated so the dashboard is full on day one; live scenarios can be generated on demand.
